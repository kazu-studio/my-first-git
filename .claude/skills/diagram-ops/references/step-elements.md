# ステップ要素の整理ガイド

手順書からHTML図解に変換する前に、各ステップを構造化する方法。

---

## ステップの分解テンプレート

手順書の各ステップを、以下の項目に分解して整理する。

| 項目 | 説明 | 書き方の例 |
|------|------|-----------|
| **ステップ番号** | 通し番号（1から始める） | `1`, `2`, `3`… |
| **アクション** | 「〜する」形式で1行に収める | `XX画面でYYボタンを押す` |
| **使用ツール** | 画面名・システム名・コマンドを具体的に | `管理画面 > ジョブ一覧`, `$ run.sh` |
| **確認事項** | 「〜であることを確認」で表現 | `ステータスが「完了」であることを確認` |
| **注意レベル** | 下表で判断 | `通常` / `注意` / `危険` |
| **分岐先** | 結果によって次ステップが変わる場合のみ記載 | `OK → Step 5` / `エラー → 担当者に連絡` |

---

## 注意レベルの判断基準

| レベル | 判断基準 | 色 |
|--------|----------|-----|
| **通常** | 失敗しても容易にやり直せる | 青 (`step-normal`) |
| **注意** | 確認を怠るとミスが発生する・分岐点がある | 橙 (`step-caution`) |
| **危険** | ミスが本番影響・データ消失・取り返しのつかない結果につながる | 赤 (`step-danger`) |

### 危険レベルの典型例

- 本番データの削除・変更
- 外部システムへの送信・連携実行
- サービスの停止・起動
- 設定ファイルの上書き

---

## アクション文の書き方

### 原則: 主語なし + 動詞で終わる

```
❌ 悪い例:
「ジョブ管理システムにアクセスして、ジョブのステータスを確認してください」

✅ 良い例:
「ジョブ管理システムでステータスを確認する」
```

### よく使う動詞パターン

| パターン | 例 |
|----------|-----|
| 画面を開く | `XX画面を開く` |
| 入力する | `YYフィールドに値を入力する` |
| 実行する | `ジョブを実行する` / `スクリプトを実行する` |
| 確認する | `ステータスをチェックする` |
| 承認・許可する | `申請を承認する` |
| 報告する | `担当者に完了を報告する` |

---

## ツール名の書き方

### 原則: 読者がどこで何をするかわかるように書く

```
❌ 悪い例:
「ツールで確認する」

✅ 良い例:
「ジョブ管理システム > ジョブ一覧 > [実行]ボタン」
「ターミナル: $ ./run.sh --env prod」
「監視ダッシュボード > アラート画面」
```

### ツール種類別の書き方

| ツール種類 | 書き方例 | Lucide icon |
|-----------|---------|-------------|
| Web画面 | `XX管理画面 > メニュー名` | `monitor` |
| ターミナル / コマンド | `$ コマンド名 オプション` | `terminal` |
| ログファイル | `ログ: /var/log/xxx.log` | `file-text` |
| データベース | `DB: テーブル名を確認` | `database` |
| 設定ファイル | `設定ファイル: config.yml` | `file-code` |
| メール / 通知 | `メール: 担当者XXへ報告` | `mail` |

---

## 確認事項の書き方

### 原則: 「〜であることを確認」で表現する

```
❌ 悪い例:
「ステータスを見る」

✅ 良い例:
「ステータスが「完了」であることを確認」
「エラーログが出ていないことを確認」
「送信件数が想定どおり（XX件）であることを確認」
```

---

## 分岐の整理方法

### 手順書の分岐を見つける

以下のような表現があれば分岐ステップ:

- 「〜の場合は〜、〜の場合は〜」
- 「正常終了したら〜」「エラーが出たら〜」
- 「成功した場合〜」「失敗した場合〜」

### 分岐カードの構造

| 分岐の種類 | 左カード | 右カード |
|-----------|---------|---------|
| 成功 / 失敗 | ✅ 正常 → Step N | ❌ エラー → エラー対応 |
| 条件A / 条件B | A の場合 → Step N | B の場合 → Step M |
| 担当者確認 | 承認 → Step N | 差し戻し → 修正して再申請 |

---

## 注意文の書き方

### 通常注意（`step-caution`）

「〜に注意する」「〜を必ず確認してから進む」

```
例:
「前のステップが完了していることを確認してから実行する」
「入力値を2回確認してから送信する」
```

### 危険警告（`step-danger` 内の赤ボックス）

「〜すると本番影響が発生する」「〜は取り返しがつかない」

```
例:
「本番データが対象。実行前に必ずバックアップを確認する」
「このコマンドは元に戻せない。対象ファイル名を必ず二重確認する」
```
